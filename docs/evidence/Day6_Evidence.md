## 3. 问题修复记录

1. **页面请求失败时原来只显示空白**：原 TradeView 等页面的 `onMounted` 直接 `await` API 调用，失败时页面显示空白。现在所有列表页均使用 try/catch/finally 结构，loading/error/data 三态完整覆盖。

2. **搜索结果为空时原来没有说明**：原来没有搜索功能，添加 SearchBar 后，通过 computed 属性配合 EmptyState 判断，搜索无结果时显示"暂无符合条件的 XXX 信息"。

3. **发布按钮重复点击风险**：PublishView 已有 `:disabled="submitting"` 和 `submitting` 状态管理，本次进一步增强了占位符提示的明确性，并添加了 `watch(publishType, resetForm)` 防止切换类型时残留字段。

4. **收藏按钮状态不明显**：原来收藏/已收藏共用 `.favorite-btn` 样式（灰色），现在增加 `.favorite-btn.active` 样式（蓝色背景 + 蓝色文字），视觉上明确区分两种状态。

5. **Store 中存在冗余代码**：清理了 `favorite.ts` 中未被任何组件调用的 `toggleFavorite` 和 `setApiRecordId` 方法。

6. **个人中心缺少页面标题**：UserCenterView 原来直接从 profile 卡片开始，缺少标题。现已添加"个人中心"页面标题和副标题。

## 7. 遇到的问题与解决方法

**问题**：优化 TradeView 时，一开始把 `loadTrades()` 中的 favorites 同步逻辑拆分到了错误处理之外，导致请求失败时 favorites 不会恢复。

**解决方法**：将 favorites 同步逻辑放入 try 块内，与数据加载保持原子性。同时确保 finally 中正确设置 `loading.value = false`，这样无论成功或失败，加载状态都能正常消失。

## 8. 今日反思

一个前端项目不能只关注"功能能跑"，还需要关注交互反馈、错误提示和用户体验。原因在于：真实用户在使用过程中会遇到各种异常情况——网络延迟、服务中断、无数据、搜索无结果、误操作等。如果没有加载状态，用户会以为页面卡死了；如果没有错误状态，接口失败后页面一片空白，用户完全不知道发生了什么；如果没有空状态提示，用户会困惑"这个页面到底有没有内容"。

今天通过为四个核心页面添加 loading/error/empty/search 四种状态，让项目的用户体验从"能用"提升到了"好用"。特别是错误状态的"重新加载"按钮，让用户可以自主恢复，而不需要刷新整个页面。收藏按钮的视觉反馈让用户清楚地知道当前是否已收藏了某条信息。

交互优化是新的一层次能力——它要求开发者站在用户的角度思考：如果我是一个普通学生，打开这个页面时，我想看到什么？如果出错了，我希望看到什么样的提示？这种同理心是写出优秀前端应用的关键。
