## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | trades | title、category、price、condition、location、description | 二手交易需要展示商品分类、价格和成色，帮助买家快速判断商品价值和状态 |
| 失物招领 | lostFounds | title、type、itemName、location、eventTime、description | 需要区分寻物和招领两种场景，物品名称和发生时间是找回失物的关键信息 |
| 拼单搭子 | groupBuys | title、type、targetCount、deadline、location、description | 目标人数和截止时间决定了拼单是否能够成团，是拼单业务的核心字段 |
| 跑腿委托 | errands | title、taskType、reward、from、to、deadline、description | 取件地点和送达地点明确了跑腿路径，酬劳和截止时间是接单方的决策依据 |

通用字段（标题、地点、描述）是所有发布类型的共同需求，在不同类型下承载不同的业务含义。

## 3. 表单校验规则

| 校验字段 | 校验规则 | 适用类型 |
|---|---|---|
| title（标题） | 不能为空 | 全部类型 |
| location（地点） | 不能为空 | 全部类型 |
| description（描述） | 不能为空 | 全部类型 |
| category（分类） | 不能为空 | 二手交易 |
| tradePrice（价格） | 必须大于 0 | 二手交易 |
| condition（成色） | 不能为空 | 二手交易 |
| itemName（物品名称） | 不能为空 | 失物招领 |
| eventTime（发生时间） | 不能为空 | 失物招领 |
| groupType（拼单类型） | 不能为空 | 拼单搭子 |
| targetCount（目标人数） | 不能少于 2 人 | 拼单搭子 |
| deadline（截止时间） | 不能为空 | 拼单搭子、跑腿委托 |
| taskType（任务类型） | 不能为空 | 跑腿委托 |
| reward（酬劳） | 不能为负数 | 跑腿委托 |
| from/src（取件地点） | 不能为空 | 跑腿委托 |
| to（送达地点） | 不能为空 | 跑腿委托 |

校验采用手动编写的 `validateForm()` 函数，未引入第三方校验库，以便于理解校验逻辑。校验失败时通过 `FormField` 组件的 `error` prop 在对应字段下方显示红色提示文字。

## 7. 遇到的问题与解决方法

**问题：** 发布类型切换后，之前填写的数据仍然保留在表单对象中，虽然界面上不显示的字段不可见，但数据不会自动清除。

**解决方法：** 在 Day4 阶段接受这种情况。对于后续优化，可以在 `watch(publishType, () => resetForm())` 中添加类型切换时自动重置表单的逻辑，但需要权衡用户体验——切换类型意味着用户更改了发布意愿，清空已填内容通常是合理的行为。

**问题：** PublishView.vue 之前只支持二手交易发布，表单字段和校验逻辑都硬编码为 trade 类型，不支持类型切换。

**解决方法：** 彻底重构了页面结构，引入 `publishType` 响应式变量控制表单字段的显示，使用 `v-if` 条件渲染不同类型专属字段，在 `validateForm()` 和 `handleSubmit()` 中分别处理四种类型的校验和提交逻辑。

## 8. 今日反思

发布表单、表单校验和数据新增是完整 Web 应用从"只读"走向"交互"的关键环节。通过 Day4 的实践，我认识到前端应用不仅要能够展示后端返回的数据，还要能够有效地收集用户输入、校验输入内容、提交数据并反馈结果。

发布表单的设计需要考虑业务类型的差异——不同业务需要不同的字段才能完整描述信息。表单校验是保证数据质量的第一道关卡，虽然 Day4 只实现了基础校验（必填、数值范围），但这已经能够过滤掉大量无效提交。数据新增则是将前端收集的用户输入转化为后端可存储的结构化数据的过程，正确的字段映射和接口路径是数据能成功写入的前提。

整个发布流程的闭环——从类型选择、字段填写、校验拦截、POST 提交到页面跳转——构成了用户交互的核心体验链路。如果这个链路不清晰，后续的用户状态管理、我的发布记录等功能就会缺乏数据基础。Day4 的设计为 Day5 的进一步开发奠定了坚实的数据流基础。
